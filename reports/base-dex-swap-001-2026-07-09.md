## Contamination, and what it cost

The first three `with_skill` runs scored 3/3 and are **excluded from every number in this report.**

For repo-shaped tasks `verify` grades `run.diff`, and `writeDiff` built that diff with a bare `git add -N .`. The skill installs to `.agents/skills/<name>/`, which is untracked and not gitignored, so the entire `SKILL.md` — including the router address `expect_2` grades — was diffed straight into the judge's evidence. The judge is specified to see neither the skill nor the variant; it saw both. `snapshotOutput` already excluded these dirs via `SKIP_DIRS`, but that is the question-shaped path, so the existing `gas` task never exposed this. Introduced with repo-shaped grading in `fdc2869`.

Patched in `scripts/verify.ts` by scoping the diff, index, and status commands to `. :(exclude).agents :(exclude).claude`, then re-running `with_skill` as runs 4–6 under new run ids. Runs 1–3 remain on disk as append-only records.

**The leak was worth one full pass.** Blind, `with_skill` is 2/3, not 3/3. Run 4 wrote a wrong router address and a sighted judge had passed the equivalent check three times out of three. Any repo-shaped `with_skill` number produced before this patch should be treated as inflated.

## What actually happened

**The headline prior barely reproduced.** The skill leads with *"The dominant DEX on each L2 is NOT Uniswap. Don't default to Uniswap on every chain."* Only one of three `no_skill` runs defaulted to Uniswap. The other two named Aerodrome unprompted. On this executor that prior is largely already corrected — the skill is not buying what its own headline claims to sell.

**What it does buy is the address.** All three `no_skill` runs failed `e2`; two of them fabricated addresses with a correct prefix and an invented tail (see `mistakes/building-blocks/fabricated-protocol-address.yaml`). `no_skill` run 3 is the sharpest case: it produced the entire correct Aerodrome call shape — `swapExactTokensForTokens`, `Route[]` with `stable: false`, the factory field — and still invented both the router and factory addresses. It knew the protocol and hallucinated the constants.

**And the skill only half-delivers the address.** `with_skill` run 4 fabricated the router *twice, differently*, with the skill open: a 40-char wrong constant in the contract, and a 39-char string — not a valid address — in its `NOTES.md`. Both share the real address's first 30 hex chars. Runs 5 and 6 copied it exactly. The likely cause is placement: the Base router and factory appear only as trailing `//` comments inside a Solidity snippet (`SKILL.md:179`, `:184`), while the "Key Protocol Addresses (Verified Mar 2026)" table is mainnet-only and carries no Aerodrome row. Addresses in a table get copied; addresses in prose get regenerated from memory.

## Skill defects found while drafting the task

Both were caught verifying expect lines against primary sources, *before* any run. Neither is a benchmark result; both are wrong in the skill as written.

1. **The merger is misdated, twice.** The skill states Aerodrome and Velodrome "merged into **Aero** (November 2025, Dromos Labs)" and that "the unified DEX dominates both Base and Optimism," in the present tense. Dromos announced the merger around January 2026 with a Q2/July 2026 launch window. As of today it is not a shipped, dominant, unified DEX. Written as settled history, this is the exact stale-prior failure the skill exists to prevent.
2. **The TVL figure is stale.** "~$500-600M TVL" for Aerodrome; DeFi Llama puts it near $323M today. The skill tells the reader to check DeFi Llama for Ethereum TVL and then quotes a hardcoded number for Base.

Neither reached the executors' output — all three blind `with_skill` runs wrote plain "Aerodrome" and never repeated the merger claim — so this cost no passes. It is latent, not observed.

`expect_1` was written to accept either brand name precisely so the grading would not depend on the skill's bad date, and `expect_4` accepts any `factory` value because Aerodrome's router falls back to `defaultFactory` when passed `address(0)` — grading the skill's hardcoded factory would have failed correct implementations.

## Harness defect: `skill_version` records the wrong repo

Every record above says `skill_version: fdc2869`. That is this repo's HEAD, not the skill's. `getSkillVersion` runs `git rev-parse` inside `skills/<name>/`, which is tracked by the framework repo, so it always reports the framework sha. The field is documented as "git short sha of the skill source." The actual source sha is `191dcc1`, recorded here only because the task `notes` say so. No record can currently tell you which revision of a skill it graded. The same applies to `skills/gas`.
